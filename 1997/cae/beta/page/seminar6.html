Die Active Template Library (ATL)

Im 6. Seminar wird die Implementierung einer COM-Klasse zur Berechnung von Querschnittswerten fortgesetzt. Das im 5. Seminar definierte Interface (interface.idl) wird kompiliert und vollständig in das Projekt eingebunden. Dafür kommt der im Developer Studio enthaltene MIDL-Kompiler zum Einsatz. Es wird eine Typenbibiliothek (Type Library) des erstellt und als Ressource hinzugefügt.

Zur Active Template Library

Die ATL ist eine Bibliothek zur Erstellung von COM-Objekten (COM-fähigen Servern), die als Steuerelemente visuell genutzt werden können oder "ohne Benutzeroberfläche" vorliegen. In Visual C++ sind zwei Assistenten zur ATL vorhanden: Der ATL COM Wizard dient der Entwicklung minimaler Komponenten, die sich zudem selbst registrieren können. Mit Hilfe des ATL Object Wizards können Objekte in vorhandene ATL-Projekte eingefügt werden.
Bei unserem Beispiel handelt es sich um einen Inproc-Server, der ohne Visualisierung implementiert werden soll. Die ATL dient hier lediglich der Implementierung von COM-Funktionen.


Automation und COM-Interfaces

In COM-Interfaces werden verschiedene Funktionen (Funktions-prototypen) zusammengefaßt und exportiert, d.h. für andere Programme zugänglich gemacht. Eine Applikation enthält üblicherweise mehrere Interfaces, in denen jeweils zusammengehörige Funktionen enhalten sind. Diese Interfaces werden als Klassen im Server implementiert. Da der Implementierungsaufwand für kleinere Server und auch für Clients enorm hoch ist, gibt es eine weitere Möglichkeit, Funktionalität für andere Programme zugänglich zu machen: Die Automation-Schnittstelle (auch OLE Automation, ActiveX-Automation genannt) ermöglicht ein relativ einfaches Exportieren von Funktionen. Hier werden alle Funktionen über ein Automation-Interface (IDispatch) aufgerufen. Soll eine Klasse Automation-fähig sein, muss sie lediglich eine Implementierung des IDispatch-Interfaces enthalten und die zu exportierenden Funktionen müssen diesem Interface bekannt sein.

Durch die ATL werden verschiedene Makros zur Implementierung des IDispatch-Interfaces angeboten, die in unserem Beispiel Verwendung finden.

Implementierung mit der ATL

  1. Starten Sie die Visual C++ Entwicklungsumgebung und laden Sie das Projekt "D:\Seminar6\processor.dsw" (auf Index-Seite herunterladen und entpacken).
  2. Fügen Sie die Dateien "interface.c" und "interface.h" dem Projekt hinzu. Diese Dateien enthalten die Deklaration des COM-Interfaces und die Definition der Indentifikatoren des Interfaces, der COM-Klasse und der Typenbibliothek.

  3. Legen Sie ein neues Ressourcen-Script mit dem Namen "ressource.rc" an (Datei-Neu).

  4. Schalten Sie den Arbeitsbereich in den Ressourcenübersicht-Modus. Doppelklicken Sie im Arbeitsbereich auf "processor Ressourcen", sodaß im rechten Fenster (Editor) processor.rc angezeigt wird. Selektieren Sie "processor.rc" und wählen Sie im Ansicht-Menü "Ressourcen-Includes...". Geben Sie unter dem Textfeld "Kompilier-Zeit-Direktiven:" 1 TYPELIB "interface.tlb" ein (Bestätigen mit OK).

    • Damit ist nun die Typenbibliothek in das Ressourcenscript eingebunden. Dies ist nötig, um der ATL die Interfacedefiniton zur Verfügung zu stellen, wodurch die Implementierung eines Automation-Interfaces möglich wird.

  5. Fügen Sie nun in die Datei "processor.h" den rot markierten Text ein.

    processor.h 
    #ifndef __processor__
#define __processor__

/////////////////////////////////////////////////////////////////////////////
// class Point2D

class Point2D : public CObject
{
  public:
  double _X;
  double _Y;
};

/////////////////////////////////////////////////////////////////////////////
// class Processor

class Processor: 
  public IDispatchImpl<IProcessor,& IID_IProcessor,& LIBID_Processor> ,
  public CComObjectRoot , public CComCoClass<Processor,& CLSID_Processor>
{
  BEGIN_COM_MAP(Processor)
    COM_INTERFACE_ENTRY(IDispatch)
    COM_INTERFACE_ENTRY(IProcessor)
  END_COM_MAP()

  DECLARE_NOT_AGGREGATABLE(Processor)

  DECLARE_REGISTRY(Processor,"Processor.Object","Processor.Object",
                   IDS_PROCESSOR_DESCRIPTION,THREADFLAGS_APARTMENT)

  // Processor members
  private:
  CObList _Geometry; 
  double _Xs,_Ys,_A,_Sx,_Sy,_Ix,_Iy,_Ixy;

  public:
  Processor();
  void Process();

  // IProcessor members
  public:
  HRESULT __stdcall AddNode(double X,double Y);
  HRESULT __stdcall GetResult(long Name,double * Result);
};

#endif

    • Durch Vererbung erhält die Klasse "Processor" COM-Funktionen, die die Bereitstellung der Funktionen und Interfaces gewährleisten.
    • Die Einträge der "COM_MAP" sind für die Macros der ATL von Bedeutung.
    • Über den Eintrag "DECLARE_REGISTRY" wird festgelegt, unter welchem Namen das Automation-Objekt registriert wird (wichtig für Aufrufe aus VBA).

/////////////////////////////////////////////////////////////////////////////
// class Processor

Processor::Processor()
{
  _Geometry.RemoveAll();
}

void Processor::Process()
{
  _A = 0.0;
  _Sx = 0.0;
  _Sy = 0.0;
  _Ix = 0.0;
  _Iy = 0.0;
  _Ixy = 0.0;

  Point2D * Point;
  double X,Y,X1,Y1;

  POSITION Position = _Geometry.GetHeadPosition();
  
  if (Position != NULL)
  {
    Point = (Point2D *) _Geometry.GetNext(Position);
    if (Point != NULL)
    {
      X1 = Point->_X - _Xs;
      Y1 = Point->_Y - _Ys;
    }
  }

  while (Position != NULL)
  {
    Point = (Point2D *) _Geometry.GetNext(Position);
    if (Point != NULL)
    {
      X = Point->_X - _Xs;
      Y = Point->_Y - _Ys;
      
      double A = (X1 * Y) - (X * Y1);
      double B = X1 + X;
      double C = Y1 + Y;
      double D = ((X1 * Y) + (X * Y1)) / 2;
      
      _A += A / 2;
      _Sx += (A * C) / 6;
      _Sy += (A * B) / 6;
      _Ix += (A * ((C * C) - (Y1 * Y))) / 12;
      _Iy += (A * ((B * B) - (X1 * X))) / 12;
      _Ixy += (A * ((B * C) - D)) / 12;
      
      X1 = X;
      Y1 = Y;
    }
  }
}

HRESULT Processor::AddNode(double X,double Y)

  Point2D * Point = new Point2D;
  Point->_X = X;
  Point->_Y = Y;
  _Geometry.AddTail(Point);
  return S_OK;

BEGIN_OBJECT_MAP(processor)

extern "C" BOOL WINAPI DllMain(HINSTANCE Instance,DWORD Reason,void *)
{
  if (Reason == DLL_PROCESS_ATTACH)
  {
    _Module.Init(processor,Instance);
	  DisableThreadLibraryCalls(Instance);
  }
  if (Reason == DLL_PROCESS_DETACH)
  {  
    _Module.Term();
  }

  return TRUE;
}
 
STDAPI DllGetClassObject(REFCLSID rclsid,REFIID riid,void ** ppv)
{
  return _Module.GetClassObject(rclsid, riid, ppv);
}

STDAPI DllCanUnloadNow()
{
  return (_Module.GetLockCount()==0) ? S_OK : S_FALSE;
}

STDAPI DllRegisterServer()
{
  return _Module.RegisterServer(TRUE);
}

STDAPI DllUnregisterServer()
{
  _Module.UnregisterServer();
	return S_OK;
}

Über Funktionen der Instanz von AtlGlobalModule "_Module" werden die Aufrufe "DllGetClassObject", "DllCanUnloadNow", "DllRegisterServer" und "DllUnRegisterServer" ebenfalls an Funktionen der Klasse "CComModule" weitergeleitet.
